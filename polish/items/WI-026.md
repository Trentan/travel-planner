## Problem

The "Reset App" button performs a full factory reset, wiping all trip data, custom cities, and file connections from `localStorage` and `IndexedDB`. Users expect "Reset" to be a less destructive action (like restarting the app or clearing UI state) and are surprised when their loaded files are "deleted" from the app's internal storage.

## Before (current state)

> Screenshot: `../screenshots/before/desktop-12-mobile-menu.png`  
> Callout: The "Reset App" button in the Actions menu and mobile menu is labeled vaguely but performs a destructive data wipe.

## Proposed fix

1. Rename "Reset App" to "Factory Reset (Wipe All Data)" in the UI to make the danger explicit.
2. Update the confirmation prompt to emphasize that the *trip data* will be lost.
3. Add a "Hard Restart" option that clears caches and reloads the app without wiping user data, addressing the need for a non-destructive technical reset.

```js
/* BEFORE */
async function resetData(options = {}) {
  // wipes everything
}

/* AFTER */
async function factoryReset(options = {}) {
  // current resetData behavior, renamed for clarity
}

async function hardRestart() {
  // clear caches + reload, keep data
}
```

## After (proposed state description)

The "Actions" menu now contains "Hard Restart" for troubleshooting and a red "Factory Reset (Wipe All Data)" at the bottom. The prompt for Factory Reset is more explicit about data loss.

## Acceptance criteria

- [ ] "Reset App" label is replaced by "Factory Reset (Wipe All Data)".
- [ ] A "Hard Restart" option is available in the menu.
- [ ] "Hard Restart" reloads the app and clears caches but preserves `localStorage` trip data.
- [ ] Regression check passes: `node scripts/regression-city-nav.js`

## How to implement

1. Modify `js/data.js` to rename `resetData` to `factoryReset` and add `hardRestart`.
2. Update `js/ui.js` to handle the new prompt and action naming.
3. Update `index.html` to reflect the new menu items.

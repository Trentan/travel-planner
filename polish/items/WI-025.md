## UX Review Findings

1. Dates are still easy to misread as secondary metadata.
   - In the current screenshots, the actual dates sit behind route labels, counters, and summary text.
   - The user has to parse several small labels before they understand where the trip currently sits in time.
   - This is the main reason the cards feel less legible than they could be.
2. The card rails are structurally consistent, but the active card still feels too tall.
   - The rail plus summary plus status plus details stack creates a lot of vertical pressure at 390px.
   - On the itinerary view, the first card in particular competes with the pager header and makes the screen feel more like a dashboard than a simple step-through trip view.
3. The layout needs a stronger trip-position tracker.
   - Right now the rail counter only says where the user is in the local rail, not where they are in the trip overall.
   - A compact tracker should show leg/day position, total trip length, and a completed/current/upcoming state.
4. There is a mild fit risk around longer content.
   - Longer property names, booking references, and multi-line route descriptions can push the summary block taller than intended.
   - That makes the visual balance inconsistent across cards and increases the chance that some screens feel clipped or overstuffed.
5. The tracker vocabulary currently implies task completion instead of position.
   - In a swipe-based surface, movement through cards is orientation, not completion.
   - "Completed" reads like an action was finished, while the user is only scrolling through the trip.
   - That mismatch is what makes the tracker feel wrong during casual scrolls.

## UX Review Findings, Follow-up

1. Day discovery is still too indirect.
   - The user has to read route labels and summary text before the actual day position becomes obvious.
   - A trip surface should surface the day first, then the date, then the route context.
   - The current treatment makes the card feel informative, but not immediately navigable.
2. The trip rail needs to stay visually bound to the active card.
   - The buttons currently act like a separate navigation strip instead of a live position indicator.
   - When the card changes, the matching chip should be brought into view and visually anchored to the active card.
   - The user should never have to guess which button matches the card they are reading.
3. Editing and navigation need clearer separation.
   - Navigation controls should move the user through the trip.
   - Edit controls should modify the current card without looking like they are part of the pager state.
   - This is especially important on mobile, where small controls are easy to misread as status indicators.
4. The current card should read as the primary surface, not the rail.
   - The rail should support orientation.
   - The card should carry the date, route, summary, and edit affordances.
   - If both compete for attention, the user loses the sense of where they are in the trip.
5. The details area is still too easy to over-expose.
   - Long metadata blocks, booking values, and activity rows can make the first visible card feel crowded.
   - The next pass should keep the summary tight and reveal deeper content only when the user asks for it.

## Interaction Guidance

- Promote `Day x` and the date to the first line of every trip card.
- Keep the trip rail sticky and auto-scroll it so the active chip stays near the card in view.
- Use a stronger active state for the current chip and quieter states for the neighboring chips.
- Let tap and swipe do the same job: both should move the user to the exact same card state.
- Keep the edit affordance on the card header or action row so users can change content without hunting through the rail.
- Collapse dense detail sections by default and reveal them only when the user expands the card.
- Preserve the shared card system across itinerary, transport, and accommodation so the navigation model stays consistent.

## Proposed Revision

The next pass should keep the shared mobile surface, but change the information architecture and the navigation model:

- Promote the date to a first-class visual element in every card header, not a supporting caption.
- Add a trip tracker bar that shows overall position, such as `Trip 4 of 18`, with clearly marked completed/current/upcoming states.
- Keep the rail counter, but make it subordinate to the trip tracker so it acts as a navigation affordance rather than the main orientation cue.
- Reduce default card height by collapsing detail sections unless the user explicitly expands them.
- Clamp the summary area so long route names and booking references do not force inconsistent card heights.
- Use a consistent date format hierarchy:
  - bold absolute date
  - smaller route or stay context
  - muted status or duration
- Recast the tracker as a positional device rather than a completion device:
  - behind instead of completed
  - here instead of current
  - ahead instead of upcoming
  - reserve completed only for explicit user toggles such as day or task checkboxes
- Consider a more distinctive visual metaphor for mobile orientation, such as a compact trip compass or beacon rail:
  - the active card becomes the lit center point
  - earlier cards fade into a restrained trail
  - future cards remain preview-only until swiped into focus
  - the tracker stays readable while scrolling because it describes position rather than achievement
- Make the matching rail chip and the current card feel mechanically linked, so the button strip reads as the map of the visible cards rather than a separate control block.
- Treat navigation and editing as different jobs: the rail moves between cards, while the card chrome edits the current card in place.
